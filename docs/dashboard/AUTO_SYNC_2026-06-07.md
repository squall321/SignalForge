## 1. 양방향 자동 동기화 개요

| 방향 | 주기 | 트리거 | 페이로드 | 평균 전송량 |
|---|---|---|---|---|
| 송신 (push) | 30분 | `sf_celery_beat` → `tasks.run_auto_sync_to_drive` | LATEST.json (~700B) + db dump delta (~50MB, sha 변동 시) + env.example (sha 변동 시) | sha 무변 cycle ≈ 1KB |
| 송신 SIF | 일 1회 21:00 UTC | beat `drive-sync-sif-daily-2100utc` | SIF 5종 (~991MB) | 일 약 1GB |
| 수신 (pull) | 5분 | cron/systemd `auto-pull.sh` | LATEST.json (~700B) fetch → sha 비교 → 변경분만 pull | 무변 cycle ≈ 700B |

핵심: **LATEST.json delta** — 수신 측은 매 cycle 700B fetch + 로컬 sha 비교, 변경 없으면 NOOP (audit `change_detected=false`). 송수신 모두 `flock` 단일 인스턴스 보장.

## 8. 자기보고 drift 명시

- `audit auto_sync.jsonl` 트랙 분포: 산출물 정의는 Y1-Y5 5트랙이나 실제 키는 `Y2` (76) / `Y4` (64) 두 가지만. Y1/Y3/Y5 작업은 build 시 `track=Y2` 또는 `track=Y4` 로 합산 기록됨 — 트랙 분리 미스. 보고/실측 차이 명시.
- `latest_manifest.available=false`: 송신 측 `sync-to-drive.sh` 가 아직 LATEST.json 을 생성하지 않음 (Y3 helper 는 만들어졌으나 sync-to-drive 본체 통합 미완). 수신 측 fallback 인 `db-dumps/` 폴더 폴링은 작동 중 (Y2 pytest T1 검증).

- send 24h ok=20/21: 실패 0건. dry_runs=19 — 거의 모두 검증 사이클.
- archive sentinel: `reports/archive/auto_sync/` 미생성. R26 패턴 미적용 — 다음 라운드 적용 권장.
