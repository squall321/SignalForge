# T5. 통합 대시보드 아키텍처 (Architecture Track)

> T1(지식 그래프) · T2(시계열 인식 변화) · T3(커뮤니티 분석) · T4(국가별 뷰)를 묶는 기술 스택, 데이터 흐름, 성능 전략, 단계적 로드맵.

---

## 1. 목적 (Why)

T1~T4는 각각 강력한 단일 뷰지만, **소비자 의사결정 워크플로우는 뷰 간 이동**(국가→커뮤니티→그래프→시계열)이 필수다. T5는 단일 뷰가 아닌 **셸(shell)** 이다.

이 트랙에서 사용자(PM·마케터·CX 리드)는 다음 의사결정을 한다.
- **어디서부터 볼지의 결정** — 글로벌 알림이 떴을 때 어떤 뷰로 진입할지 (랜딩의 KPI 4종이 시작점)
- **뷰 간 컨텍스트 유지** — 한국·갤럭시 S25·최근 7일이라는 필터가 4 뷰 모두에 자동 전파되는가
- **신뢰 가능한 응답속도** — 11만→100만 행으로 가도 200ms를 유지하는가
- **누구에게 공유 가능한가** — 임원/협력사에게 URL만으로 같은 뷰가 보이는가 (셰어 URL = 필터 상태 직렬화)

T5가 실패하면 T1~T4는 각각 좋은 데모지만 통합된 제품이 되지 못한다.

## 2. 데이터 모델 (공통 레이어)

T1~T4가 공통으로 의존하는 **3개 materialized view** 와 **1개 Redis 캐시 키 스킴**.

```sql
-- (1) 핵심 일자별 집계 (모든 뷰의 출발점)
CREATE MATERIALIZED VIEW mv_voc_daily AS
SELECT
  date_trunc('day', published_at)::date AS day,
  product_id,
  platform_id,
  country_code,
  COUNT(*)                       AS voc_count,
  AVG(sentiment_score)           AS sentiment_avg,
  SUM(engagement_score)          AS engagement_sum,
  COUNT(*) FILTER (WHERE sentiment_label='negative') AS neg_count,
  COUNT(*) FILTER (WHERE sentiment_label='positive') AS pos_count
FROM voc_records
WHERE published_at >= NOW() - INTERVAL '365 days'
GROUP BY 1,2,3,4;

CREATE UNIQUE INDEX ON mv_voc_daily(day, product_id, platform_id, country_code);
CREATE INDEX ON mv_voc_daily(product_id, day DESC);

-- (2) 카테고리 폭발 뷰 (T1 그래프 엣지, T2 카테고리 트렌드)
CREATE MATERIALIZED VIEW mv_voc_category_daily AS
SELECT
  date_trunc('day', published_at)::date AS day,
  product_id, platform_id, country_code,
  unnest(categories)         AS category,
  COUNT(*)                   AS n,
  AVG(sentiment_score)       AS sentiment_avg
FROM voc_records
WHERE published_at >= NOW() - INTERVAL '180 days'
  AND array_length(categories,1) > 0
GROUP BY 1,2,3,4,5;

-- (3) 커뮤니티 헬스 (T3)
CREATE MATERIALIZED VIEW mv_platform_health AS ...;

-- REFRESH 전략: Celery beat 30분 주기 CONCURRENTLY refresh
```

**Redis 캐시 키 스킴**: `sf:{view}:{product}:{period}:{country}:{platform}:hash` (TTL 5분, 신규 VOC broadcast 시 invalidate 패턴 `sf:*:{product}:*`).

## 3. 프론트 스택 결정

| 후보 | 장점 | 단점 | 판정 |
|---|---|---|---|
| **React + Vite + AntD** | 차트 라이브러리 풍부(echarts/recharts/visx), AntD Table·Filter 즉시 사용, 빌드 1.5초 | SSR 없음 (SEO 불필요하므로 OK) | **채택** |
| Next.js 14 (App Router) | SSR, RSC | 내부 대시보드라 SSR 가치 작음, RSC가 차트 SDK와 충돌 | 보류 |
| SvelteKit | 번들 작음, 빠름 | 팀 학습비용, AntD급 컴포넌트 부족 | 탈락 |

**근거**: 대시보드는 ① 인증된 사용자만 접근 ② 검색엔진 노출 불필요 ③ 데이터 헤비(차트 30+)이므로 클라이언트 사이드 SPA가 자연스럽다. Next.js의 RSC가 차트 라이브러리(브라우저 의존)와 호환되지 않아 오히려 부담.

**채택 패키지**: React 18 + Vite 5 + TypeScript 5 + AntD 5 + **TanStack Query v5** + **ECharts** (지도/그래프/시계열 모두 커버) + **Cytoscape.js** (T1 그래프 인터랙션) + zustand (필터 글로벌 상태) + dayjs + axios.

**TanStack Query 정책**: `staleTime: 60s`, `gcTime: 5min`, refetch on window focus = false (대시보드 백그라운드 탭에서 폭주 방지), WebSocket new_voc 수신 시 `queryClient.invalidateQueries(['voc', productId])`.

## 5. WebSocket 실시간 활성 계획

현재 `/ws/realtime` endpoint는 존재하나 **broadcast_new_voc 호출부가 비어 있음**. 활성화 단계:

1. **Celery worker hook** — `crawler/tasks.py::save_voc()` 마지막에 `redis.publish("voc:new", json)` 호출 (Celery → FastAPI 직접 호출은 프로세스 분리상 불가).
2. **FastAPI listener** — startup 시 Redis pub/sub 구독 task 시작, 메시지 수신 → `manager.broadcast()`.
3. **클라이언트 throttle** — 24시간 13k건 = 평균 9건/분, 피크 시 50+/분 가능. 클라이언트는 1초 버킷으로 디바운스 후 UI 갱신 (개별 카드 깜빡임 방지).
4. **메시지 페이로드** — `{type, voc_id, product_code, platform_code, country_code, sentiment, published_at}` (가벼움 유지, 전체 본문 X — 필요 시 별도 fetch).

## 8. 성능 전략 (200ms @ 100만 행)

| 레이어 | 전략 | 효과 |
|---|---|---|
| PG | mv_voc_daily 일자 집계로 raw 11만 → 9k 행 | 100배 축소 |
| PG | `CREATE INDEX ON voc_records (product_id, published_at DESC, sentiment_label)` 복합 | drill 쿼리 50ms |
| PG | `categories` GIN 인덱스 | 카테고리 필터 30ms |
| API | Redis 캐시 5분 TTL + key invalidation pub/sub | hit율 70%+ 예상 |
| API | FastAPI gzip middleware (compresslevel=6) | 페이로드 70% 감소 |
| API | response_model `exclude_none=True` | JSON 크기 20% 감소 |
| 프론트 | RQ staleTime 60초 + structuralSharing | 동일 페이지 재방문 0ms |
| 프론트 | ECharts `lazyUpdate` + virtual scroll(테이블) | 렌더 16ms 유지 |
| 프론트 | Code splitting per page (Vite dynamic import) | 초기 JS < 300KB |

**100만 행 임계점**: mv refresh가 30분 내 끝나지 않으면 partitioning 도입 (`voc_records_2026_06`). Phase 4에서 평가.

## 9. 인증/접근제어 결정

**결정: 내부 전용 + 임시 공유 링크.**

- Phase 1~3: 사내망 + Nginx basic auth (단순).
- Phase 4: OAuth(Google Workspace) + role 2단계 (`viewer`, `admin` — admin은 알림 룰 편집).
- 공유 링크: 필터 URL + 30일 만료 토큰 (`/share/{token}` → 읽기 전용 뷰, 새 알림 룰 작성 불가).

외부 공개는 명시적 거절 — VOC 본문에 개인 식별 텍스트 포함 가능성.

## 10. 배포

- **빌드**: Vite build → `dist/` 정적 자산.
- **서빙**: 기존 Nginx에 location `/dashboard/` 추가 → `try_files $uri /dashboard/index.html`.
- **API 프록시**: Nginx `/api/`, `/ws/` → backend:8000 (이미 구성).
- **CI**: GitHub Actions → SSH rsync `dist/` → `/var/www/signalforge/dashboard/`. 무중단 (정적 파일이라 reload 불필요).
- **Vercel은 채택 안 함**: 백엔드가 사내망에 있어 CORS·인증 복잡도 증가. 단일 서버 단일 도메인이 단순.

## 14. 트레이드오프와 한계

- **mv refresh 지연**: CONCURRENTLY 30분 주기 → 최신 30분 VOC는 mv 미반영. 실시간 KPI는 raw 테이블 fallback(작은 윈도우만) — 코드 복잡도 ↑.
- **WebSocket 스케일**: 단일 FastAPI 프로세스 ConnectionManager는 다중 워커 시 broadcast 누락. uvicorn worker 1개 유지 또는 Redis pub/sub broker 도입 필요.
- **AntD 무게**: 번들 3MB+. Tree-shaking으로 1.2MB까지 줄여도 모바일 첫 로딩 부담 → 모바일 Overview만 노출로 우회.
- **차트 라이브러리 단일화**: ECharts 1종으로 통일은 학습비용 ↓ 이지만 그래프 인터랙션이 약함 → Cytoscape 추가로 절충 (2종 유지).
- **인증 미흡**: Phase 1 basic auth는 진짜 보안이 아님. P4 OAuth 전까지는 사내망 IP 화이트리스트 필수.

## 15. 검증 기준

1. **응답시간**: Overview/Timeline/Geo의 p95 응답 ≤ 200ms (Locust 100 동시 사용자 부하, mv 워밍 후).
2. **공유성**: 임의 필터 조합 URL을 새 브라우저(쿠키 없는 incognito + basic auth)에서 열어 동일 데이터 로드 — 시각적 동등 (스크린샷 diff < 2%).
3. **실시간성**: 새 VOC가 PG에 INSERT된 시각 ~ 대시보드 Overview KPI count 증가 시각 ≤ 5초 (P4 종료 기준).
